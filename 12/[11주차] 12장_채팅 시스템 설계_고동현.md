# [11주차] 12장_채팅 시스템 설계_고동현

## 1단계: 문제 이해 및 설계 범위 확정

설계할 시스템의 요구사항을 다음과 같이 가정합니다.

- **지원 기능:** 응답지연이 낮은 1:1 채팅, 최대 100명까지 참여 가능한 그룹 채팅.
- **부가 기능:** 사용자의 접속상태 표시, 다중 단말 지원(여러 기기 동시 접속), 푸시 알림.
- **트래픽 규모:** 5천만 DAU (일간 능동 사용자 수).
- **메시지 제한:** 텍스트 메시지만 지원하며, 길이는 100,000자 이하.
- **보관 기간:** 채팅 이력은 영구 보관.

## 2단계: 개략적 설계안 제시 및 동의 구하기

채팅 시스템은 클라이언트끼리 직접 통신하지 않으며, 중간에 채팅 서비스가 메시지를 받아 수신자에게 전달하는 역할을 합니다.

### 1. 네트워크 통신 프로토콜

- **HTTP:** 초기 연결이나 무상태 요청(로그인 등)에 사용하며, Keep-alive 헤더를 통해 연결을 유지하면 효율적입니다. 하지만 서버가 클라이언트에게 임의로 메시지를 보내기 어렵습니다.
- **폴링(Polling) / 롱 폴링(Long Polling):** 클라이언트가 주기적으로 새 메시지를 묻거나 대기하는 방식이나, 서버 자원이 낭비되는 단점이 있습니다.
- **🌟 웹소켓(WebSocket):** 서버가 클라이언트에게 비동기 메시지를 보낼 때 널리 쓰이는 기술로, 양방향 영구 연결을 제공하여 실시간 채팅에 가장 적합합니다.

### 2. 개략적 아키텍처

시스템은 크게 세 부분으로 나뉩니다.

- **무상태(Stateless) 서비스:** 로그인, 회원가입, 사용자 프로필 등을 처리하는 전통적인 API 서버들입니다. 로드밸런서 뒤에 위치하며 **서비스 탐색(Service Discovery)** 기능도 이곳에 포함됩니다.
- **상태 유지(Stateful) 서비스:** 각 클라이언트와 웹소켓으로 독립적인 네트워크 연결을 유지하는 '채팅 서비스'입니다.
- **제3자 서비스 연동:** 앱이 실행 중이 아닐 때도 알림을 받기 위한 '푸시 알림' 서비스입니다.

### 3. 데이터 모델 및 저장소

- **일반 데이터:** 사용자 프로필, 설정, 친구 목록 등은 안정적인 관계형 데이터베이스(RDBMS)에 보관합니다.
- **채팅 이력 (Chat History):** 읽기/쓰기 비율이 1:1에 달하고 데이터 양이 엄청나므로, 수평적 규모 확장이 쉽고 데이터 접근 지연시간이 낮은 **키-값 저장소(Key-Value Store, NoSQL)**를 사용합니다.
- **메시지 ID:** `message_id`는 고유해야 하며 시간 순서대로 정렬 가능해야 합니다. 7장의 스노플레이크(Snowflake)를 쓰거나, 같은 채널 안에서만 순서를 보장하면 되는 점을 이용해 **지역적 순서 번호 생성기(Local sequence number generator)**를 활용할 수 있습니다.

## 3단계: 상세 설계

가장 중요한 핵심 컴포넌트들의 상세 동작 원리입니다.

### 1. 서비스 탐색 (Service Discovery)

- 클라이언트에게 위치(Geographical location), 서버 용량 등을 고려하여 가장 적합한 채팅 서버를 찾아주는 역할을 합니다.
- 오픈 소스인 **아파치 주키퍼(Apache Zookeeper)**를 주로 사용하여, 클라이언트가 접속을 시도하면 사전에 정한 기준에 따라 최적의 채팅 서버를 골라줍니다.

### 2. 메시지 처리 흐름

- **1:1 채팅 흐름:**
    1. 사용자 A가 채팅 서버 1로 메시지를 전송합니다.
    2. ID 생성기가 `message_id`를 결정하고, 메시지가 키-값 저장소에 저장됩니다.
    3. 사용자 B가 접속 중이라면 채팅 서버 2(B가 연결된 서버)로 전송되고, 오프라인이라면 푸시 알림 서버로 전달됩니다.
- **여러 단말 간의 동기화:** 각 단말은 `cur_max_message_id`라는 변수를 유지하여, 키-값 저장소에 있는 메시지 중 이 ID보다 큰 최신 메시지만 가져와 동기화합니다.
- **소규모 그룹 채팅:** 사용자 A가 보낸 메시지를 그룹에 속한 사용자(B, C 등)의 메시지 동기화 큐(Inbox)에 각각 복사(Fan-out)합니다. 소규모 그룹(예: 위챗의 500명 제한)에서는 이 방법이 빠르고 효율적입니다.

### 3. 접속상태 표시 (Presence Server)

- **로그인/로그아웃:** 클라이언트가 웹소켓 연결을 맺거나 끊을 때 키-값 저장소에 상태와 타임스탬프를 기록합니다.
- **접속 장애 (네트워크 단절):** 인터넷 연결이 불안정할 때마다 상태를 바꾸면 서버에 큰 부담이 됩니다. 클라이언트가 주기적으로 **박동(Heartbeat) 이벤트**를 보내도록 하고, 특정 시간(예: 30초) 동안 박동이 없으면 오프라인 상태로 변경합니다.
- **상태 정보 전송:** 발행-구독(Pub-Sub) 모델을 사용하여, 사용자의 상태가 변경되면 친구들의 채널로 변경 사실을 발행합니다. 단, 그룹 크기가 클 경우 입장하는 순간에만 상태를 읽어오도록 최적화해야 합니다.

## 4단계: 마무리 (추가 논의 포인트)

면접의 마지막에 시간이 남는다면 다음과 같은 확장성 및 안정성 문제를 추가로 논의해 볼 수 있습니다.

- **미디어 파일 지원:** 사진이나 비디오는 용량이 크므로 압축 방식, 클라우드 저장소 활용, 썸네일 생성 등을 논의할 수 있습니다.
- **종단 간 암호화(End-to-End Encryption):** 발신인과 수신자 외에는 서버조차 메시지 내용을 볼 수 없도록 하는 보안 기술입니다.
- **클라이언트 측 캐시:** 이미 읽은 메시지를 단말에 캐시해 두면 서버와 주고받는 데이터 양을 크게 줄일 수 있습니다.
- **오류 처리:** 채팅 서버에 장애가 나면 주키퍼가 클라이언트에게 새로운 서버를 배정하여 재접속하도록 하고, 메시지 전송에 실패하면 큐(Queue)를 이용해 재시도(Retry)하는 메커니즘을 둡니다.
