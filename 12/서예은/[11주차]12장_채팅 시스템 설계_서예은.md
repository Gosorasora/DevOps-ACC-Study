# 12장 채팅 시스템 설계

채팅 시스템은 시장에 다양한 형태로 존재한다. WhatsApp, Facebook Messenger, WeChat, Line, Google Hangout, Discord 등이 대표적이다. 설계에 들어가기 전에 요구사항을 명확히 하지 않으면 면접관이 기대한 것(예: 1:1)과 다른 것(예: 그룹 전용)을 설계하는 실수를 할 수 있다.

---

# 1. 문제 이해 및 설계 범위

## 1.1 면접 Q&A로 확정한 요구사항

| 구분 | 내용 |
| --- | --- |
| 채팅 유형 | 1:1 채팅 + 그룹 채팅 (그룹 최대 100명) |
| 플랫폼 | 모바일 + 웹 |
| 규모 | DAU 약 5천만 명 |
| 메시지 | 텍스트 중심 (문자 수 제한 예: 100,000자) |
| 접속 상태 | 온라인/오프라인 등 presence 표시 |

## 1.2 기능 요구사항 (Functional)

- 1:1·그룹 메시징
- 사용자 접속 상태( presence ) 표시
- 저지연 1:1 실시간 대화
- 멀티 디바이스 동시 로그인
- 푸시 알림

## 1.3 비기능 요구사항 (Non-functional)

- 대규모 트래픽(수천만 DAU) 처리
- 모바일·웹 동시 지원
- 확장 가능한 아키텍처

## 1.4 이번 설계에서 제외·후술 가능한 항목

- 종단간 암호화(E2E): 당장 필수는 아니며, 시간이 남으면 논의
- 미디어(사진·동영상) 첨부: 텍스트만 전제로 시작

# 2. 개략적 설계안

## 2.1 채팅 서비스의 역할

클라이언트는 서로 직접 통신하지 않고 **중앙의 채팅 서비스**를 경유한다.

1. 클라이언트로부터 메시지 수신
2. 수신자를 판별하고 메시지 전달
3. 오프라인 수신자를 위해 메시지 저장 후 재연결 시 전달

## 2.2 송신·수신 프로토콜

### 클라이언트 → 서버 (메시지 전송)

- **HTTP**가 널리 쓰인다.
- **Keep-Alive**로 연결을 유지하면 TCP 핸드셰이크 오버헤드를 줄일 수 있다.

### 서버 → 클라이언트 (메시지 수신·푸시)

HTTP는 클라이언트가 요청을 시작하는 모델이라 서버 푸시가 어렵다. 대표적인 보완 기법:

| 방식 | 요약 |
| --- | --- |
| 폴링(Polling) | 주기적으로 “새 메시지 있나?” 요청 |
| 롱 폴링(Long Polling) | 응답이 있을 때까지(또는 타임아웃까지) 연결 유지 |
| WebSocket | 양방향 지속 연결 |

### 개념 흐름

메시지 송신 클라이언트 → **채팅 서비스**(메시지 저장 + 메시지 전달) → 메시지 수신 클라이언트

---

# 3. 폴링과 롱 폴링

## 3.1 폴링

- 클라이언트가 일정 간격으로 서버에 “새 메시지가 있나?” 요청한다.
- **문제**: 폴링 주기가 짧을수록 비용·부하 증가, 새 메시지가 없을 때도 서버 자원이 낭비된다.

## 3.2 롱 폴링

- 요청 후 새 메시지가 생기거나 **타임아웃**될 때까지 연결을 유지한다.
- 메시지를 받으면 연결을 닫고, 곧바로 다음 요청을 보내 반복한다.

**약점**

1. **로드밸런싱**: 송신자와 수신자가 다른 채팅 서버에 붙어 있으면, 메시지를 받은 서버에 수신자의 롱 폴링 연결이 없을 수 있다(라운드 로빈 등과 궁합 이슈).
2. **연결 끊김 감지**: 클라이언트가 조용히 끊긴 경우 서버가 알기 어렵다.

---

# 4. WebSocket

## 4.1 특징

- 서버가 클라이언트에 비동기로 메시지를 보내는 데 가장 흔히 쓰인다.
- **HTTP 핸드셰이크**(예: GET) 후 서버 ACK, 연결을 **WebSocket**으로 승격 → **지속적·양방향** 통신.
- HTTP/HTTPS와 동일한 **80/443** 포트를 사용해 방화벽 친화적이다.
- 메시지 빈도가 낮은 클라이언트에 대해 무작정 폴링하는 것보다 효율적일 수 있다.

## 4.2 채팅에서의 사용

- 송신·수신 모두 동일 프로토콜로 단순하다.
- 연결이 **지속**되므로 서버는 연결 관리를 효율적으로 해야 한다.

## 4.3 개략 설계에서의 역할

- 실시간 채널은 **WebSocket**, 그 외 비실시간 API는 **HTTP**를 병행하는 구성이 흔하다.

---

# 5. 무상태·상태 유지·제3자 연동

시스템을 크게 세 덩어리로 나눌 수 있다.

## 5.1 무상태 서비스 (Stateless)

- 로그인, 회원가입, 사용자 프로필 등 전통적인 요청/응답 처리.
- 로드밸런서 뒤에 두는 일반적인 웹/모바일 백엔드 패턴.
- 예: **서비스 탐색**, **인증**, **그룹 관리**, **사용자 프로필** 등.
- 구현은 모놀리식이든 마이크로서비스든 가능하다.

### 서비스 탐색(Service Discovery)

- 클라이언트가 접속할 **채팅 서버의 호스트명(DNS)** 등을 알려 주는 역할이 핵심이다.

## 5.2 상태 유지 서비스 (Stateful)

- 이 설계에서는 **채팅 서비스**가 지속 연결을 유지해야 하므로 상태가 필요하다.
- 서버가 살아 있는 한 클라이언트는 보통 다른 서버로 옮기지 않는다.

## 5.3 제3자 서비스 연동

- **푸시 알림**: 앱이 꺼져 있을 때도 메시지를 알리는 데 필수에 가깝다. (10장 알림 시스템 설계와 연결)

## 5.4 규모 확장성

- 저트래픽이면 이론상 한 대에 모든 기능을 올릴 수 있으나, 대규모에서는 부적절하다.
- 예: 동시 접속 100만 명, 연결당 메모리 10KB 가정 시 약 **10GB** RAM — 단일 서버로도 “숫자”는 가능해 보일 수 있으나, **단일 장애점(SPOF)** 때문에 실제 설계에서는 분산·이중화가 중요하다.
- 면접에서는 단일 서버 개념 설계로 시작해 점진적으로 분산 구조로 다듬는 전개가 유효하다.

# 6. 상세 아키텍처

| 구성 요소 | 역할 |
| --- | --- |
| 사용자 단말 | HTTP(로드밸런서 경유), WebSocket(실시간 서비스) |
| API 서버 | 로그인, 가입, 프로필 변경 등 비실시간 |
| 실시간 서비스 | **채팅 서버**(메시지 중계), **접속상태 서버**(presence) |
| 알림 서버 | 푸시 알림 전송 |
| 키-값 저장소 | 채팅 이력 등 저장 — 과거 대화 조회 |

- 클라이언트는 채팅 서버와 **WebSocket**으로 지속 연결을 유지한다.

# 7. 저장소(Storage)

## 7.1 데이터 유형별 선택

| 유형 | 예시 | 권장 |
| --- | --- | --- |
| 일반 데이터 | 프로필, 설정, 친구 목록 | **RDB** — 안정성, **복제·샤딩**으로 확장 |
| 채팅 이력 | 대량·최근 조회 위주·가끔 검색/점프 | **키-값 저장소** |

## 7.2 채팅 이력의 특징

- 양이 매우 많다(예: 하루 수십억 건 수준의 서비스 언급).
- **최근 메시지** 조회가 대부분, 오래된 메시지는 상대적으로 적게 본다.
- 검색, 멘션, 특정 메시지로 이동 등 **임의 접근**이 필요할 수 있다.
- 1:1 채팅은 읽기·쓰기 비율이 대략 **1:1**에 가깝다.

## 7.3 키-값 저장소를 쓰는 이유

- **수평 확장**이 용이하다.
- 지연 시간을 낮추기 유리하고, RDB보다 “긴 꼬리” 데이터에 잘 맞는다는 설명이 나온다.

**실사례 (책 예시)**

- Facebook Messenger: HBase
- Discord: Cassandra

# 8. 데이터 모델

## 8.1 1:1 채팅 메시지

- PK: `message_id` (순서 유지)

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| message_id | bigint | PK |
| message_from | bigint | 발신 |
| message_to | bigint | 수신 |
| content | text | 본문 |
| created_at | timestamp | 생성 시각 |

## 8.2 그룹 채팅 메시지

- 복합 PK: `(channel_id, message_id)`
- 파티션 키: `channel_id`

| 컬럼 | 타입 |
| --- | --- |
| channel_id | bigint (PK/파티션) |
| message_id | bigint (PK) |
| message_to | bigint |
| content | text |
| created_at | timestamp |

# 9. 메시지 ID

요구 조건:

1. **전역 유일성**
2. **시간 순서 반영**: 새 ID가 이전보다 커서 정렬에 사용 가능

생성 방식 예:

- **Snowflake**: 전역 64비트 시퀀스
- **로컬 시퀀스**: 특정 1:1 세션·채널 안에서만 유일하면 되므로 구현이 단순할 수 있음

---

# 10. 상세 설계 — 서비스 탐색

역할: 지리적 위치, 서버 부하 등을 고려해 클라이언트에 **적합한 채팅 서버**를 추천한다.

- Zookeeper 등으로 “사용 가능한 서버 목록”을 유지하는 **디렉터리** 역할을 하는 경우가 많다.

## 흐름 요약

1. 사용자 A 로그인 시도
2. 로드밸런서 → API 서버
3. API 서버 인증 후 **서비스 탐색**(예: Zookeeper)으로 최적 채팅 서버(예: 채팅 서버 2) 조회
4. 사용자 A가 해당 서버와 **WebSocket** 연결

---

# 11. 메시지 흐름 (1:1)

1. 사용자 A → 채팅 서버 1로 메시지 전송
2. 채팅 서버 1 → **ID 생성기**로 `message_id` 획득
3. **메시지 동기화 큐**로 전달
4. **키-값 저장소**에 저장
5. 사용자 B 상태에 따라 분기  
   - **온라인**: B가 붙어 있는 채팅 서버(예: 채팅 서버 2)로 전달  
   - **오프라인**: **푸시 알림 서버**로 전달
6. 온라인인 경우 채팅 서버 2가 B에게 WebSocket으로 전달

---

# 12. 멀티 디바이스 동기화

- 각 단말은 **`cur_max_message_id`**(해당 기기에서 마지막으로 받은 메시지 ID)를 유지한다.
- 키-값 저장소의 메시지 중 **수신자 ID가 나**이고, **`message_id` > `cur_max_message_id`** 이면 “새 메시지”로 동기화 대상이 된다.
- 한 사용자가 폰·노트북 등 여러 세션을 동시에 열어도, 기기별로 최신 ID만 추적하면 된다.

---

# 13. 소규모 그룹 채팅

- 그룹에 메시지를 보낼 때, **그룹원마다 별도의 메시지 동기화 큐**에 복사한다.
- **장점**: 각 사용자는 “내 큐”만 보면 되어 구현이 단순하다.
- **소규모 그룹**에서는 복사 비용이 작다. WeChat은 이 방식을 쓰며 그룹 규모를 제한(예: 500명)하는 식으로 언급된다.
- **대규모 그룹**에서는 메시지를 모든 멤버 큐에 복사하는 비용이 커지므로 다른 전략이 필요하다.

여러 채팅 서버가 동일 수신자의 **메시지 동기화 큐**로 밀어 넣고, 수신자가 소비하는 형태로도 설명된다.

# 14. 접속상태(Presence)

## 14.1 역할

- 이름 옆 초록 점 등 **온라인/오프라인** 표시.
- **접속상태 서버**가 WebSocket으로 실시간 처리한다.

## 14.2 로그인

- WebSocket 연결 후 접속상태 서버가 키-값 저장소에 예: `status: online`, `last_active_at` 기록.

## 14.3 로그아웃

- API 서버 등을 통해 로그아웃 시 키-값 저장소에서 `offline`으로 갱신.

## 14.4 접속 장애·박동(Heartbeat)

- 단순히 “끊기면 오프라인”으로 두면 터널 등에서 상태가 깜빡인다.
- 클라이언트가 **주기적으로 박동 이벤트**를 보내고, 서버는 마지막 박동 후 **x초** 이내면 온라인으로 유지한다. 예시로 **5초마다 박동**, **30초** 동안 미수신 시 오프라인 전환 같은 수치가 나온다.

## 14.5 상태 정보 전파 (Pub-Sub)

- 친구마다 채널(예: A-B, A-C)을 두고, 친구가 **구독**해 WebSocket으로 상태 변경을 받는다.
- **아주 큰 그룹**(예: 10만 명)에는 모든 이에게 브로드캐스트하는 것은 비효율적이어서, 채팅방 입장 시에만 갱신하거나 수동 새로고침 등이 대안으로 제시된다.

# 15. 4단계 마무리

## 15.1 요약

- 1:1·그룹 채팅, 실시간은 **WebSocket** 중심.
- 핵심 구성: **채팅 서버**, **접속상태 서버**, **푸시 알림 서버**, **키-값 저장소(이력)**, **API 서버(기타 기능)**.

## 15.2 추가로 논의할 수 있는 주제

| 주제 | 내용 |
| --- | --- |
| 미디어 | 사진·동영상은 용량이 커서 압축, 객체 스토리지, 썸네일 등 |
| E2E 암호화 | WhatsApp처럼 송수신자만 읽도록 하는 방식 |
| 캐싱 | 클라이언트에 읽은 메시지 캐시로 트래픽 절감 |
| 로딩 속도 | Slack처럼 지역별로 사용자·채널 데이터를 분산하는 사례 |
| 장애 대응 | 채팅 서버 다운 시 Zookeeper 등으로 재배치 |
| 재전송 | 재시도·큐로 신뢰성 보강 |
